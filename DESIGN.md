# Spotify Track Console Notifier - Дизайн-док

## 1. Цель

Сделать локальное приложение для одного пользователя, которое отслеживает воспроизведение в Spotify и при смене трека выводит в консоль:

`Исполнитель - Название трека`

Это MVP только для личного локального использования (без мультипользовательского режима, деплоя и UI).

## 2. Объем работ

### Входит в MVP

- OAuth-авторизация Spotify для одного пользователя.
- Периодический опрос состояния воспроизведения.
- Определение смены трека через сравнение текущего и предыдущего идентификаторов.
- Вывод строки в консоль при старте нового трека.
- Обновление access token без перезапуска приложения.
- Базовая устойчивость к ошибкам API и лимитам.

### Не входит в MVP

- GUI/веб-интерфейс.
- Постоянная история/база данных.
- Поддержка нескольких аккаунтов.
- Уведомления вне терминала (desktop/mobile/webhooks).
- Аналитика, очереди ретраев, продвинутая наблюдаемость.

## 3. Поведение продукта

### Основной пользовательский сценарий

Как локальный пользователь, когда я переключаюсь на новую песню в Spotify, я хочу в течение нескольких секунд увидеть в терминале исполнителя и название.

### Функциональные требования

1. Приложение аутентифицируется в Spotify Web API через Authorization Code flow.
2. Приложение непрерывно опрашивает endpoint текущего воспроизведения с настраиваемым интервалом.
3. Если текущий трек отличается от последнего выведенного, приложение печатает одну строку в консоль.
4. Один и тот же трек не должен печататься повторно, пока он не сменился.
5. При истечении токена приложение автоматически обновляет его через refresh token.
6. Состояния «ничего не играет» и «пауза» обрабатываются без шумных логов.

### Нефункциональные требования

1. Выход в режим мониторинга после успешной авторизации: <= 10 секунд.
2. Целевая задержка обнаружения смены трека: 2-4 секунды (зависит от интервала опроса).
3. Стабильная работа в долгом локальном запуске (часы).
4. Минимум зависимостей и простая настройка.

## 4. Ограничения Spotify API

1. В Spotify Web API нет realtime webhook для событий смены трека.
2. Поэтому нужен polling (периодический опрос).
3. Релевантные scopes:
- `user-read-currently-playing`
- `user-read-playback-state` (рекомендуется как fallback и для более полного состояния)
4. Есть rate limit; приложение должно обрабатывать `429 Too Many Requests` через `Retry-After`.

## 5. Предлагаемая архитектура

Небольшой слоистый CLI-сервис:

1. `Config`
2. `AuthManager`
3. `SpotifyClient`
4. `TrackWatcher`
5. `ConsoleNotifier`
6. `App` (сборка компонентов + жизненный цикл)

### 5.1 Config

Ответственность:

- Читать и валидировать переменные окружения.
- Предоставлять типизированный runtime-конфиг.

Входные параметры:

- `SPOTIFY_CLIENT_ID`
- `SPOTIFY_CLIENT_SECRET`
- `SPOTIFY_REDIRECT_URI`
- `POLL_INTERVAL_MS` (по умолчанию: `2500`)
- `PRINT_ON_START` (по умолчанию: `true`)
- `TOKEN_STORAGE_PATH` (по умолчанию: `.spotify-tokens.json`)

### 5.2 AuthManager

Ответственность:

- Выполнять начальный OAuth-логин.
- Сохранять и загружать токены локально.
- Обновлять access token при истечении или при `401`.

Модель данных:

- `accessToken: string`
- `refreshToken: string`
- `expiresAtEpochMs: number`

Поведение:

1. При старте читать файл токенов.
2. Если валидного токена нет, запускать auth flow (открыть браузер + локальный callback).
3. Перед вызовами API проверять актуальность токена.
4. При `401` обновлять токен и делать один повтор запроса.

### 5.3 SpotifyClient

Ответственность:

- Инкапсулировать HTTP-вызовы Spotify.
- Преобразовывать сырой JSON в внутреннюю нормализованную модель.

Основная операция:

- `getCurrentlyPlaying(): PlaybackSnapshot | null`

`PlaybackSnapshot`:

- `isPlaying: boolean`
- `itemType: "track" | "episode" | "ad" | "unknown"`
- `trackId: string | null`
- `trackName: string | null`
- `artists: string[]`
- `progressMs: number | null`
- `fetchedAtEpochMs: number`

Стратегия endpoint:

1. Основной: `GET /v1/me/player/currently-playing`
2. Опциональный fallback: `GET /v1/me/player` при необходимости более широкого состояния

### 5.4 TrackWatcher

Ответственность:

- Периодически получать текущее состояние воспроизведения.
- Решать, является ли это событием «новый трек».

Состояние:

- `lastReportedTrackId: string | null`
- `lastItemType: string | null`

Правила решения (MVP):

1. Если активного трека нет -> ничего не делать.
2. Если тип item не `track` -> ничего не делать.
3. Если `isPlaying` равно `false` -> ничего не делать.
4. Если `trackId` отличается от `lastReportedTrackId` -> эмитить `NewTrackEvent`.
5. Обновлять `lastReportedTrackId` только после успешного эмита.

Правило старта:

- Если `PRINT_ON_START=false`, на первом успешном опросе инициализировать `lastReportedTrackId` текущим треком.
- Если `PRINT_ON_START=true`, сразу эмитить текущий трек на первом опросе.

### 5.5 ConsoleNotifier

Ответственность:

- Печатать нормализованную строку в консоль.

Формат:

- По умолчанию: `Artist1, Artist2 - Track Name`
- Опционально с временем: `[HH:mm:ss] Artist - Track`

### 5.6 App (оркестратор)

Ответственность:

- Связывать компоненты.
- Управлять запуском/остановкой.
- Обрабатывать фатальные ошибки и сигналы завершения.

Поток выполнения:

1. Загрузить конфиг.
2. Инициализировать авторизацию.
3. Запустить таймер watcher.
4. На каждый `NewTrackEvent` вызывать notifier.
5. На `SIGINT/SIGTERM` останавливать таймер и завершаться корректно.

## 6. Машина состояний (watcher)

Состояния:

1. `INIT`
2. `READY`
3. `BACKOFF`
4. `STOPPED`

Переходы:

1. `INIT -> READY` после успешной авторизации и первичной настройки опроса.
2. `READY -> READY` на штатном тике.
3. `READY -> BACKOFF` при `429` или временной сетевой ошибке.
4. `BACKOFF -> READY` после задержки.
5. Любое состояние -> `STOPPED` по сигналу завершения.

## 7. Стратегия обработки ошибок

### HTTP/API ошибки

1. `401 Unauthorized`
- Обновить токен.
- Повторить запрос один раз.
- Если повтор снова неуспешен: залогировать ошибку и перейти к следующему тику.

2. `429 Too Many Requests`
- Прочитать `Retry-After`.
- Приостановить опрос на это время (+ небольшой jitter).

3. `5xx` или сетевой timeout
- Логировать компактное предупреждение.
- Продолжать опрос на следующем тике.

### Локальные ошибки

1. Поврежденный файл токенов
- Показать предупреждение и запустить новый auth flow.
2. Отсутствуют обязательные env-переменные
- Падать на старте с понятным сообщением.

## 8. Безопасность и приватность

1. Хранить credentials в локальных переменных окружения, а не в коде.
2. Хранить файл токенов локально с максимально ограниченными правами доступа.
3. Не логировать access/refresh токены.
4. Приложение локальное; внешняя экспозиция не нужна, кроме loopback callback для OAuth.

## 9. Значения по умолчанию

- `POLL_INTERVAL_MS=2500`
- `PRINT_ON_START=true`
- `TOKEN_STORAGE_PATH=.spotify-tokens.json`
- HTTP timeout: `5000 ms`
- Количество повторов после refresh: `1`

## 10. Наблюдаемость (минимальная)

Категории логов:

1. `INFO`: старт, завершение авторизации, запуск watcher.
2. `TRACK`: выведенная строка о треке.
3. `WARN`: временные проблемы API/сети/лимитов.
4. `ERROR`: невосстановимые ошибки старта.

Принцип логирования:

- Логи короткие; без шумного debug-лога на каждом тике по умолчанию.

## 11. План тестирования (до/во время реализации)

### Ручные приемочные проверки

1. Запуск приложения, когда Spotify уже играет:
- Ожидаемо: либо мгновенный вывод, либо «тихий» старт в зависимости от `PRINT_ON_START`.

2. Переключение на другую песню:
- Ожидаемо: одна новая строка в консоли в пределах интервала опроса.

3. Пауза воспроизведения:
- Ожидаемо: новой строки о треке нет.

4. Возобновление той же песни:
- Ожидаемо: дубля строки нет.

5. Переключение на следующий трек:
- Ожидаемо: ровно одна строка для нового трека.

6. Истечение access token:
- Ожидаемо: приложение автоматически обновляет токен и продолжает работу.

7. Сценарий rate limit (или мок `429`):
- Ожидаемо: приложение уходит в backoff и затем восстанавливается.

### Цели unit-тестов

1. Логика определения смены трека.
2. Поведение на старте для `PRINT_ON_START=true/false`.
3. Нормализация ответов Spotify.
4. Ветка refresh/retry при `401`.
5. Расчет backoff при `429`.

## 12. Риски и меры

1. Пропуск смены трека из-за лага API:
- Мера: подобрать интервал polling (2-3с).

2. Дубли вывода на переходных состояниях:
- Мера: строгое сравнение `trackId` и правило одноразового эмита.

3. Инвалидация токена пользователем:
- Мера: корректный отказ и повторный запуск auth flow.

4. Особые случаи устройства/контента (реклама, подкасты, локальные файлы):
- Мера: игнорировать неподдерживаемый `itemType` в MVP.

2. Реализовать OAuth и хранение токенов.
3. Реализовать Spotify-клиент и нормализованную модель snapshot.
4. Реализовать движок принятия решения в watcher.
5. Реализовать console notifier и wiring жизненного цикла.
6. Добавить точечные unit-тесты критичной логики.
7. Провести ручную валидацию по приемочным сценариям.

## 14. Критерии готовности (MVP)

1. На реальном Spotify-аккаунте при смене трека в терминале появляется `Artist - Title` в течение 2-4 секунд.
2. Нет дублирующихся выводов для неизменившегося трека.
3. Приложение переживает истечение токена через refresh.
4. Состояния «ничего не играет/пауза» не приводят к падению и лог-спаму.
5. Инструкций по локальному запуску достаточно для использования.
