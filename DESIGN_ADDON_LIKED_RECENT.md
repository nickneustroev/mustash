# DESIGN ADDON: Auto Playlists for Recent Liked Tracks

## 1. Цель

Добавить автоматические плейлисты, которые отражают самые последние треки из `Liked Songs` (избранного), с несколькими размерами окон.

Начальная конфигурация:

1. 20 треков
2. 50 треков
3. 100 треков

Требование:

1. Плейлисты обновляются автоматически, когда пользователь добавляет трек в избранное или удаляет его.
2. Количество и размеры окон должны быть гибко настраиваемыми через env.

## 2. Ключевые ограничения Spotify API

1. Нет webhook-событий на изменения `Liked Songs`.
2. Поэтому нужен polling (`GET /v1/me/tracks`).
3. Для удаления из избранного корректнее делать полную синхронизацию содержимого целевых плейлистов, а не точечные update.

## 3. Конфигурация

Новые env-параметры:

1. `LIKED_RECENT_ENABLED=true`
2. `LIKED_RECENT_WINDOWS=20,50,100`
3. `LIKED_RECENT_PLAYLIST_PREFIX=LIKED RECENT`
4. `LIKED_RECENT_PLAYLIST_SUFFIX=[AUTO]`
5. `LIKED_RECENT_SYNC_INTERVAL_MS=15000`
6. `LIKED_RECENT_PLAYLIST_PRIVATE=true`

Правила валидации:

1. `LIKED_RECENT_WINDOWS` -> массив положительных целых.
2. Удалять дубликаты, сортировать по возрастанию.
3. Максимум окна ограничить `<= 1000` (практический предел для polling страницами).
4. Пустой список окон при `ENABLED=true` -> ошибка конфигурации.

## 4. Именование плейлистов

Формула имени:

`{LIKED_RECENT_PLAYLIST_PREFIX} {N} {LIKED_RECENT_PLAYLIST_SUFFIX}`

Пример:

1. `LIKED RECENT 20 [AUTO]`
2. `LIKED RECENT 50 [AUTO]`
3. `LIKED RECENT 100 [AUTO]`

## 5. Архитектура

### 5.1 `LikedTracksService`

Ответственность:

1. Загружать последние liked-треки по `added_at`.
2. Возвращать список URI в порядке newest -> oldest.

API:

1. `getRecentLikedUris(limit: number): Promise<string[]>`

Реализация:

1. `GET /v1/me/tracks?limit=50&offset=...`
2. Пагинация до достижения `limit`.
3. Игнорировать элементы без валидного `track.uri`.

### 5.2 `LikedRecentPlaylistManager`

Ответственность:

1. Для каждого окна `N` находить/создавать соответствующий плейлист.
2. Делать полную замену треков в плейлисте на первые `N` из liked-списка.

API:

1. `ensurePlaylists(windows: number[]): Promise<Map<number, string>>`
2. `syncWindow(playlistId: string, uris: string[]): Promise<void>`

### 5.3 `LikedRecentSyncScheduler`

Ответственность:

1. Периодически запускать sync.
2. Снижать лишние записи в Spotify (не писать, если состав окна не изменился).
3. Обрабатывать `401`, `429`, сетевые ошибки.

API:

1. `start()`
2. `stop()`
3. `syncNow()`

## 6. Алгоритм синхронизации

На каждом цикле:

1. Берем окна из конфига, например `[20, 50, 100]`.
2. Вычисляем `maxWindow` (здесь `100`).
3. Загружаем последние liked URI длиной `maxWindow`.
4. Для каждого окна `N` формируем `windowUris = likedUris.slice(0, N)`.
5. Сравниваем с последним синхронизированным снимком (локальный кэш в памяти/файле).
6. Если изменилось -> `PUT /v1/playlists/{id}/tracks` с полным списком.
7. Если не изменилось -> пропускаем запись.

Почему полная замена:

1. Корректно учитывает и добавление, и удаление из избранного.
2. Избегает сложных дифф-операций.

## 7. Состояние и кеш

Локальный state-файл (опционально, но рекомендуется), например `.liked-recent-state.json`:

1. `windows: number[]`
2. `playlistIdsByWindow: Record<number, string>`
3. `lastHashesByWindow: Record<number, string>`
4. `updatedAtEpochMs: number`

Назначение:

1. Уменьшить лишние update-запросы.
2. Быстрее восстановиться после рестарта.

## 8. Ошибки и устойчивость

1. `401` -> refresh token + retry.
2. `429` -> backoff по `Retry-After`.
3. Сетевая ошибка -> warn и retry на следующем цикле.
4. Ошибка одного окна/плейлиста не должна останавливать весь sync-цикл.

## 9. Необходимые scopes

1. `user-library-read`
2. `ugc-image-upload` (для загрузки обложки при создании плейлиста)
3. `playlist-read-private`
4. `playlist-modify-private`

Опционально:

1. `playlist-modify-public` при необходимости публичных плейлистов.

## 10. Тест-кейсы

1. Парсинг `LIKED_RECENT_WINDOWS` (`"20,50,100"` -> `[20,50,100]`).
2. Удаление дублей и сортировка (`"100,20,20,50"` -> `[20,50,100]`).
3. Формирование имени плейлиста для окна.
4. При добавлении в liked новый трек попадает в начало всех окон.
5. При удалении из liked трек исчезает из всех окон.
6. При неизменном liked-списке лишний `PUT` не выполняется.
7. При `429` sync корректно откладывается.

## 11. Критерии готовности

1. Автоплейлисты для всех окон из env существуют и синхронизируются.
2. Добавление в избранное отражается в плейлистах в пределах `LIKED_RECENT_SYNC_INTERVAL_MS`.
3. Удаление из избранного отражается так же.
4. Конфигурация окон полностью управляется env без изменений кода.
5. При создании плейлиста один раз устанавливается обложка `N recent`.
