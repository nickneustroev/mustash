# ТЗ: Auto Playlist `LIKED ARCHIVE`

## 1. Цель

Добавить автоматический плейлист `LIKED ARCHIVE [AUTO]`, в который попадает трек, если он был в `Liked Songs` и перестал быть избранным.

Ключевое бизнес-правило:

1. Если трек был лайкнут ранее и на очередной проверке больше не найден в `Liked Songs`, он должен быть добавлен в `LIKED ARCHIVE [AUTO]`.

## 2. Ожидаемое поведение

1. Плейлист `LIKED ARCHIVE [AUTO]` создается автоматически (если отсутствует).
2. Добавление в архив происходит автоматически без ручных действий пользователя.
3. Фича работает параллельно с `LIKED RECENT` и не ломает ее.
4. Поведение идемпотентное: один и тот же трек не должен добавляться в архив повторно, если он уже там.

## 3. Ограничения Spotify API

1. Нет webhook-события об удалении трека из `Liked Songs`.
2. Детекция возможна только через polling (`/v1/me/tracks`) и/или batch-проверку (`/v1/me/tracks/contains`).
3. Лимиты API требуют бережного подхода (batch до 50 ID, обработка `429`).

## 4. Термины

1. `Known liked set` - локально сохраненный набор track ID, которые на последней успешной синхронизации считались лайкнутыми.
2. `Archived set` - локально сохраненный набор track ID, уже добавленных в `LIKED ARCHIVE [AUTO]`.
3. `Unlike event` - факт, что track ID был в `Known liked set`, но больше не найден в `Liked Songs`.

## 5. Конфигурация

Новые env-параметры:

1. `LIKED_ARCHIVE_ENABLED=true|false` (по умолчанию `false`)
2. `LIKED_ARCHIVE_PLAYLIST_NAME=LIKED ARCHIVE [AUTO]`
3. `LIKED_ARCHIVE_SYNC_INTERVAL_MS=60000` (минимум 15000)
4. `LIKED_ARCHIVE_PLAYLIST_PRIVATE=true|false` (по умолчанию `true`)
5. `LIKED_ARCHIVE_STATE_PATH=.liked-archive-state.json`
6. `LIKED_ARCHIVE_BOOTSTRAP_MODE=baseline|backfill` (по умолчанию `baseline`)

Правила:

1. При `LIKED_ARCHIVE_ENABLED=true` все параметры валидируются на старте.
2. `baseline`: первый запуск только формирует baseline без массового добавления старых удалений.
3. `backfill`: первый запуск трактует все отсутствующие из baseline как кандидаты в архив (может дать большой объем записей).

## 6. Состояние

Локальный state-файл `LIKED_ARCHIVE_STATE_PATH`:

```json
{
  "version": 1,
  "playlistId": "string",
  "knownLikedById": {
    "trackId": {
      "uri": "spotify:track:...",
      "lastSeenAtEpochMs": 0
    }
  },
  "archivedIds": ["trackId"],
  "scanCursor": {
    "offset": 0,
    "updatedAtEpochMs": 0
  },
  "updatedAtEpochMs": 0
}
```

Требования к state:

1. Атомарная запись (через temp-file + rename), чтобы избежать порчи файла.
2. Толерантность к отсутствию/повреждению файла: лог warning и пересоздание.

## 7. Архитектура

### 7.1 `LikedArchiveSyncService`

Ответственность:

1. Запуск по таймеру.
2. Обнаружение `unlike event`.
3. Добавление треков в архивный плейлист.
4. Обновление локального state.

API:

1. `start()`
2. `stop()`
3. `syncNow()`

### 7.2 Расширение `SpotifyClient`

Новые/дополненные методы:

1. `getSavedTracksPage(limit: number, offset: number)` -> элементы с `track.id` и `track.uri`.
2. `checkSavedTracksContains(ids: string[])` -> `boolean[]` для batch-проверки.
3. `addPlaylistItems(playlistId: string, uris: string[])` -> добавление в конец батчами по 100.
4. `getPlaylistTrackIds(playlistId: string)` -> при инициализации `archivedIds` (опционально, один раз).

## 8. Алгоритм синхронизации

На каждом цикле:

1. Убедиться, что `LIKED ARCHIVE [AUTO]` существует (`find-or-create`).
2. Загрузить state.
3. Обновить `Known liked set`:
- прочитать первые страницы `Liked Songs` (новые лайки всегда в начале);
- обновить `knownLikedById` для найденных ID/URI;
- периодически прогонять `checkSavedTracksContains` по уже известным ID, чтобы находить удаления глубоко в библиотеке;
- стратегия прохода по всему `knownLikedById` должна гарантировать eventual coverage (любой трек будет перепроверен).
4. Для ID, где `contains=false`, сформировать кандидатов `removedIds`.
5. Отфильтровать `removedIds`, исключив уже присутствующие в `archivedIds`.
6. Для оставшихся ID собрать URI из state и добавить в `LIKED ARCHIVE [AUTO]` батчами.
7. Добавленные ID внести в `archivedIds`.
8. Удалить `removedIds` из `knownLikedById`.
9. Сохранить state.

Гарантии алгоритма:

1. Любой снятый лайк будет обнаружен и заархивирован (eventual consistency).
2. Нет повторного добавления уже архивированного трека.

## 9. Политика порядка в плейлисте

1. Новые архивные треки добавляются в конец плейлиста (append-only).
2. Порядок отражает время обнаружения удаления из избранного.

## 10. Ошибки и устойчивость

1. `401` -> refresh token + retry 1 раз.
2. `429` -> лог + отложить обработку до следующего цикла (с учетом `Retry-After`).
3. Сетевые ошибки -> warning, текущий цикл считается неуспешным, state не портится.
4. Ошибка одного batch не должна приводить к потере уже обработанных данных.

## 11. Логирование

Минимальные события:

1. Старт/стоп сервиса.
2. Создание/обнаружение архивного плейлиста.
3. Количество обнаруженных `removedIds` за цикл.
4. Количество реально добавленных в архив за цикл.
5. Причины пропуска (дубликат в `archivedIds`, нет URI).
6. Ошибки API и retry-решения.

## 12. Тест-кейсы

1. Первый запуск в режиме `baseline` не делает массовое архивирование.
2. Трек был в liked, затем removed -> добавлен в `LIKED ARCHIVE [AUTO]`.
3. Повторный unlike того же трека не создает дубликат в архиве.
4. Частичное падение при batch-add не повреждает state.
5. Корректная обработка `401` и `429`.
6. Поврежденный state-файл пересоздается и сервис продолжает работу.
7. После рестарта сервиса ранее архивированные ID не добавляются заново.
8. Detected removal из "старой" части библиотеки тоже попадает в архив (не только из top-N).

## 13. Критерии приемки

1. При снятии лайка с любого ранее известного трека он попадает в `LIKED ARCHIVE [AUTO]` автоматически.
2. Дубликаты в архивном плейлисте отсутствуют (по track ID).
3. Сервис стабильно работает при длительном запуске и перезапусках.
4. При ошибках Spotify API процесс восстанавливается без ручного вмешательства.
5. Фича включается/выключается только через env.

## 14. Что не входит в scope

1. UI для просмотра/управления архивом.
2. Мульти-аккаунт режим.
3. Гарантированный real-time (только polling с eventual consistency).
