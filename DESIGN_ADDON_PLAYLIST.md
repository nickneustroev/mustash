# DESIGN ADDON: Auto Playlist `HISTORY [AUTO]`

## 1. Цель

Расширить текущее приложение так, чтобы последние прослушанные треки автоматически поддерживались в отдельном плейлисте Spotify:

- Название плейлиста: `HISTORY [AUTO]`
- Размер окна: `N=100`
- Дубликаты: разрешены (`allow duplicates`)
- Источник добора пропусков: `Recently Played` backfill

## 2. Ожидаемое поведение

1. При каждом новом треке (событие из `TrackWatcher`) трек добавляется в локальную историю.
2. История хранит только последние 100 элементов.
3. Плейлист `HISTORY [AUTO]` регулярно синхронизируется с локальной историей.
4. Если приложение было выключено или пропустило события, история добирается из `GET /v1/me/player/recently-played`.
5. В плейлисте допускаются одинаковые треки, если они реально слушались несколько раз.

## 3. Scope и допущения

### Входит

- Автосоздание/поиск плейлиста `HISTORY [AUTO]`.
- Локальное хранение состояния истории.
- Синхронизация плейлиста по принципу "последние 100".
- Backfill из recently played.

### Не входит

- Поддержка нескольких пользователей.
- Сложные merge-стратегии с ручными правками плейлиста пользователем.
- Полная историческая библиотека (больше 100).

## 4. API и scopes

Необходимые Spotify scopes:

1. `user-read-currently-playing`
2. `user-read-playback-state`
3. `user-read-recently-played`
4. `playlist-modify-private`
5. `playlist-read-private` (для поиска/проверки целевого плейлиста)

Ключевые endpoint'ы:

1. `GET /v1/me` - получить `userId`.
2. `GET /v1/me/playlists` - найти `HISTORY [AUTO]`.
3. `POST /v1/users/{user_id}/playlists` - создать плейлист при отсутствии.
4. `GET /v1/me/player/recently-played` - backfill.
5. `PUT /v1/playlists/{playlist_id}/tracks` - полностью заменить треки в плейлисте (до 100 URI за вызов).

## 5. Компоненты

### 5.1 `PlaylistManager`

Ответственность:

1. На старте найти или создать `HISTORY [AUTO]`.
2. Держать `playlistId` в памяти.
3. Выполнять полную синхронизацию треков в плейлист.

Методы:

1. `ensurePlaylist(): Promise<string>`
2. `replaceItems(trackUris: string[]): Promise<void>`

### 5.2 `HistoryStore`

Ответственность:

1. Хранить rolling-историю из 100 URI локально.
2. Обновлять историю при новых событиях.
3. Сериализовать/десериализовать состояние в файл.

Формат состояния:

1. `items: string[]` (порядок: newest -> oldest)
2. `updatedAtEpochMs: number`
3. `lastBackfillCursor?: string` (опционально для оптимизации)

### 5.3 `BackfillService`

Ответственность:

1. Периодически запрашивать recently played.
2. Добавлять отсутствующие записи в локальную историю.
3. Не ломать порядок "новые сверху".

Принцип:

1. Получить список recently played (обычно newest -> older).
2. Вставлять элементы в историю в правильном порядке времени прослушивания.
3. После вставки обрезать до 100.

### 5.4 `PlaylistSyncScheduler`

Ответственность:

1. Не дергать Spotify API на каждый трек мгновенно.
2. Делать debounce-синхронизацию (например, каждые 5-10 секунд при изменениях).
3. Гарантировать sync при graceful shutdown.

## 6. Алгоритм данных

## 6.1 Нормализация записи истории

Источник:

1. Текущее воспроизведение (`TrackWatcher`).
2. Recently played (`BackfillService`).

Нормализованная запись:

1. `trackUri` (например, `spotify:track:...`)
2. `playedAtEpochMs`
3. `source: "live" | "backfill"`

Правила:

1. Если URI отсутствует или item не `track` -> пропуск.
2. Локальные файлы/недоступные объекты без URI -> пропуск.

## 6.2 Правила rolling-окна (N=100)

1. Новые записи вставляются в голову списка.
2. Дубликаты не удаляются.
3. После каждой вставки: `items = items.slice(0, 100)`.

## 6.3 Синхронизация в плейлист

1. Брать `items` из `HistoryStore`.
2. Вызывать `PUT /playlists/{id}/tracks` с полным массивом URI.
3. Один запрос покрывает до 100 треков, что соответствует `N=100`.

Плюсы полного `PUT`:

1. Гарантированная консистентность "как в локальном состоянии".
2. Нет сложной логики точечного удаления/добавления.

## 7. Порядок обработки событий

1. `TrackWatcher` фиксирует новый трек.
2. `HistoryStore.add(trackUri)` обновляет локальное окно.
3. `PlaylistSyncScheduler.markDirty()` ставит отложенную синхронизацию.
4. Параллельно по таймеру работает `BackfillService`.
5. Если backfill внес изменения -> снова `markDirty()`.
6. По истечению debounce -> `PlaylistManager.replaceItems(history.items)`.

## 8. Backfill-стратегия

Рекомендуемый режим:

1. Полный backfill на старте приложения.
2. Далее периодический backfill, например, раз в 60 секунд.
3. Дополнительно backfill после восстановления из rate-limit/backoff.

Источники дублей при allow duplicates:

1. Один и тот же трек в разные моменты времени должен остаться дублем.
2. В пределах одного backfill-ответа возможны повторы, их сохраняем как есть.

## 9. Обработка ошибок

1. `401` -> refresh token + retry.
2. `429` -> backoff по `Retry-After`.
3. Ошибка записи локального state -> `ERROR`, но процесс продолжает live-режим.
4. Ошибка sync плейлиста -> повтор на следующем цикле scheduler.
5. Ошибка backfill -> лог `WARN`, не блокирует live-события.

## 10. Конфигурация (новые env)

1. `HISTORY_PLAYLIST_NAME=HISTORY [AUTO]`
2. `HISTORY_MAX_ITEMS=100`
3. `HISTORY_STATE_PATH=.history-state.json`
4. `PLAYLIST_SYNC_DEBOUNCE_MS=7000`
5. `BACKFILL_INTERVAL_MS=60000`
6. `BACKFILL_LIMIT=50`

Примечание:

1. `HISTORY_MAX_ITEMS` держать `<=100`, чтобы sync был одним `PUT`.

## 11. Тест-кейсы

1. Добавление нового live-трека -> запись в историю + sync.
2. 101-й трек вытесняет самый старый.
3. Дубликаты трека сохраняются в истории и в плейлисте.
4. Backfill после простоя добавляет пропущенные треки.
5. При `429` sync откладывается и затем успешно восстанавливается.
6. Если плейлист не найден, он создается автоматически.
7. Если пользователь вручную изменил плейлист, следующая sync возвращает состояние к rolling-history.

## 12. Критерии готовности

1. Плейлист `HISTORY [AUTO]` существует (создается автоматически при отсутствии).
2. После серии прослушиваний плейлист содержит ровно последние 100 прослушанных треков.
3. Повторы треков отображаются как отдельные элементы.
4. После перезапуска приложения и backfill история остается непрерывной.
5. Нет постоянного спама API (debounce и backoff работают).
